# 实验5 进程通信

## 一、课程设计题目

实验5 进程通信

## 二、程序功能及设计思路

### 2.1 程序功能

本实验实现了三种进程间通信（IPC）方式：
1. **管道通信**：父子进程之间的管道通信
2. **消息队列通信**：Client-Server 进程间的 System V 消息队列通信
3. **共享内存通信**：两个进程通过 POSIX 共享内存通信

主程序提供菜单选择，用户可以选择不同的通信方式进行测试。

### 2.2 三种IPC机制的原理和特点

**1. 管道通信**
- 原理：内核中维护的缓冲区，数据单向流动
- 特点：简单、半双工、只能用于父子/兄弟进程、数据流式
- 优点：实现简单，无需额外同步机制
- 缺点：容量有限，只能单向传输

**2. 消息队列通信**
- 原理：内核中维护的消息链表，每个消息有类型
- 特点：可随机访问（按类型接收）、全双工、独立于进程
- 优点：可以按类型接收消息，无需同步
- 缺点：数据有大小限制

**3. 共享内存通信**
- 原理：多个进程共享同一块物理内存区域
- 特点：最快的IPC方式、需要额外同步机制
- 优点：速度快，数据量大
- 缺点：需要同步（如信号量）

### 2.3 三种通信方式的适用场景对比

| 特性 | 管道 | 消息队列 | 共享内存 |
|------|------|---------|---------|
| 速度 | 中等 | 较慢 | 最快 |
| 复杂度 | 简单 | 中等 | 中等 |
| 同步 | 自动 | 自动 | 需要手动 |
| 双向 | 半双工 | 全双工 | 全双工 |
| 亲缘进程 | 是 | 否 | 否 |
| 适用场景 | 简单数据流 | 结构化消息 | 大数据量 |

## 三、数据结构及算法设计

### 3.1 管道通信的数据流向

```
子进程 → write → 管道 → read → 父进程
```

子进程发送自己的PID和消息内容，父进程读取并显示。

### 3.2 消息队列的通信流程

使用 System V IPC 的 `msgget()`、`msgsnd()`、`msgrcv()` 系统调用：

```c
#define SVKEY 75   // 消息队列关键字
#define REQ   1    // 请求消息类型

struct msgbuf {
    long mtype;         // 消息类型
    int  sender_pid;    // 发送方进程PID
    char text[100];     // 消息正文
};
```

**通信流程**（Server/Client 模型，通过 fork 创建子进程模拟）：

```
Server(父进程)                    Client(子进程)
    │                                │
    ├─ msgget(SVKEY, IPC_CREAT)      │
    │                                ├─ msgget(SVKEY, 0)
    │                                ├─ msgsnd(REQ, self_pid)
    ├─ msgrcv(REQ) ←─────────────────┤
    ├─ 显示 "serving for client"     │
    ├─ msgsnd(client_pid, server)    │
    │                ───────────────→├─ msgrcv(self_pid)
    │                                ├─ 显示 "receive reply from"
    ├─ msgctl(IPC_RMID)              │
```

关键点：
- Server 等待类型为 REQ(=1) 的消息
- Client 发送类型为 REQ 的请求，附上自己的 PID
- Server 使用 Client 的 PID 作为应答消息类型，确保只有对应 Client 能收到
- Server 最后删除消息队列，释放内核资源

### 3.3 共享内存的创建、附加、读写、删除流程

使用 POSIX 共享内存 (`shm_open` + `mmap`)，可在 macOS 和 Linux 上通用：

```c
// 1. 创建共享内存对象
int fd = shm_open("/os_shm", O_CREAT | O_RDWR, 0666);

// 2. 设置大小
ftruncate(fd, 512);

// 3. 映射到进程地址空间
void *ptr = mmap(NULL, 512, PROT_READ | PROT_WRITE,
                 MAP_SHARED, fd, 0);

// 4. 读写（直接内存访问）
strcpy((char *)ptr, "Hello");

// 5. 取消映射
munmap(ptr, 512);

// 6. 删除共享内存对象
shm_unlink("/os_shm");
```

## 四、程序运行情况

### 4.1 管道通信运行结果

```
===== 管道通信 =====
[子进程 PID=12345] 发送: PID 12345 is sending a message to parent
[父进程 PID=12344] 接收: PID 12345 is sending a message to parent
```

### 4.2 消息队列通信运行结果（Linux）

```
===== 消息队列通信 (System V IPC) =====
[Server] 创建/获取消息队列, ID=0
[Server PID=12344] serving for client 12345
[Server PID=12344] 发送应答给 client 12345
[Client PID=12345] 发送请求消息
[Client PID=12345] receive reply from 12344
[Server] 消息队列已删除
```

**消息队列使用 `ipcs` 命令查看**：
```bash
$ ipcs -q    # 查看系统中的消息队列
$ ipcrm -q <msqid>   # 手动删除消息队列（程序异常退出时备用）
```

### 4.3 共享内存通信运行结果

```
===== 共享内存通信 (POSIX) =====
[进程A PID=12344] 初始写入: Shared memory initialized by Process A!
[进程B PID=12345] 从共享内存读到: Shared memory initialized by Process A!
[进程B PID=12345] 写入共享内存: Hello from Process B!
[进程A PID=12344] 从共享内存读到: Hello from Process B!
```

### 4.4 三种IPC机制对比

| 通信方式 | 通信方向 | 同步方式 | 数据模型 | 适用场景 |
|---------|---------|---------|---------|---------|
| 管道 | 单向 | 自动阻塞 | 字节流 | 父子进程简单通信 |
| 消息队列 | 全双工 | 按类型阻塞 | 结构化消息 | Client-Server 通信 |
| 共享内存 | 全双工 | 需手动同步 | 直接内存读写 | 大数据量高效传输 |

## 五、编程中遇到的困难及解决方法

### 5.1 遇到的困难

1. **消息队列的跨平台兼容性**：
   - System V 消息队列（msgget/msgsnd/msgrcv）在 Linux 上完整支持
   - macOS 不支持完整的 System V 消息队列
   - 需要使用条件编译 `#ifdef __linux__` 区分平台

2. **消息队列的清理**：
   - 程序异常退出时消息队列可能残留在内核中
   - 需要使用 `msgctl(IPC_RMID)` 或在命令行用 `ipcrm` 手动清理

3. **管道读写阻塞**：
   - 管道读操作默认阻塞，需要确保写入端正确关闭
   - 否则读操作会一直等待不会返回

4. **共享内存同步**：
   - 父子进程访问共享内存需要协调（使用 wait/sleep 简化）

### 5.2 解决方法

1. **平台兼容**：
   - Linux 上：直接使用 System V 消息队列
   - macOS 上：通过条件编译打印说明，并用 pipe 模拟演示输出
   - 建议在 Linux 虚拟机上进行完整测试

2. **消息队列清理**：
   - Server 进程在通信结束后调用 `msgctl(msqid, IPC_RMID, NULL)`
   - 残留在内核中的队列可用 `ipcrm -q <msqid>` 或 `ipcrm -Q <key>` 删除

3. **管道管理**：
   - 仔细管理管道文件描述符，不需要的一端立即关闭
   - 父进程 wait() 确保子进程结束

4. **共享内存同步**：
   - 父进程中 fork 后调用 wait(NULL) 等待子进程完成操作
   - 实际应用中应使用信号量等同步原语

### 5.3 实习心得

通过本次实验，深入理解了三种主要 IPC 机制的工作原理和适用场景。

**管道**是最简单的 IPC 方式，适合父子进程间的简单数据传输，但只能单向且只能用于有亲缘关系的进程。

**消息队列**提供了更灵活的通信方式，支持按消息类型接收，实现了 Client-Server 通信模型。但 System V 消息队列需要手动管理生命周期，异常退出时需要清理。

**共享内存**是最快的 IPC 方式，进程直接访问同一块内存，但需要额外的同步机制来避免竞态条件。

三种 IPC 机制各有优缺点，实际开发中需要根据具体需求选择合适的方式：
- 简单快速的数据传递 → 管道
- 结构化消息通信 → 消息队列
- 大数据量高效传输 → 共享内存

Mac 与 Linux 的 IPC 差异让我认识到可移植性的重要性。POSIX IPC 比 System V IPC 更现代，跨平台性更好。建议在 Linux 系统上进行完整测试。
