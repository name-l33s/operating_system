# 实验5 进程通信

## 一、课程设计题目

实验5 进程通信

## 二、程序功能及设计思路

### 2.1 程序功能

本实验实现了三种进程间通信（IPC）方式：
1. 管道通信
2. 消息队列通信（演示）
3. 共享内存通信

主程序提供菜单选择，用户可以选择不同的通信方式进行测试。

### 2.2 三种IPC机制的原理和特点

**1. 管道通信**
- 原理：内核中维护的缓冲区，数据单向流动
- 特点：简单、半双工、只能用于父子/兄弟进程、数据流式
- 优点：实现简单，无需额外同步机制
- 缺点：容量有限，只能单向传输

**2. 消息队列通信**
- 原理：内核中维护的消息链表，每个消息有类型
- 特点：可随机访问（按类型）、全双工、独立于进程
- 优点：可以按类型接收消息，无需同步
- 缺点：数据有大小限制，系统范围内

**3. 共享内存通信**
- 原理：多个进程共享同一块物理内存区域
- 特点：最快的IPC方式、需要额外同步机制
- 优点：速度快，数据量大
- 缺点：需要同步（如信号量），需要注意同步问题

### 2.3 三种通信方式的适用场景和区别

| 特性 | 管道 | 消息队列 | 共享内存 |
|------|------|---------|---------|
| 速度 | 中等 | 较慢 | 最快 |
| 复杂度 | 简单 | 中等 | 中等 |
| 同步 | 自动 | 自动 | 需要手动 |
| 双向 | 半双工 | 全双工 | 全双工 |
| 关系进程 | 是 | 否 | 否 |
| 适用场景 | 简单数据流 | 结构化消息 | 大数据量 |

## 三、数据结构及算法设计

### 3.1 管道通信的数据流向

```
子进程 → write → 管道 → read → 父进程
```

子进程发送自己的PID和消息内容，父进程读取并显示。

### 3.2 消息队列的消息结构定义

（Linux System V IPC标准）

```c
struct msgbuf {
    long mtype;         // 消息类型，必须大于0
    char mtext[100];    // 消息内容
};

// 使用方式：
// msgget(key, flags) - 创建消息队列
// msgsnd(msqid, msgp, msgsz, msgflg) - 发送
// msgrcv(msqid, msgp, msgsz, msgtyp, msgflg) - 接收
```

通信流程：
1. Server创建消息队列
2. Client发送类型为1的请求（包含PID）
3. Server接收请求
4. Server发送应答（类型为Client PID）
5. Client接收应答

### 3.3 共享内存的创建、附加、读写、删除流程

```c
// 1. 创建
shm_open(name, O_CREAT | O_RDWR, mode)
// 2. 设置大小
ftruncate(fd, size)
// 3. 附加到地址空间
mmap(addr, length, prot, flags, fd, offset)
// 4. 读写
直接访问映射的内存区域
// 5. 分离
munmap(addr, length)
// 6. 删除
shm_unlink(name)
```

## 四、程序运行情况

### 4.1 管道通信运行结果

```
=== 管道通信 ===
[子进程] 发送: PID 1234 is sending a message to parent
[父进程] 接收: PID 1234 is sending a message to parent
```

### 4.2 消息队列通信（演示）

```
=== 消息队列通信 (演示) ===
注意: Mac系统的System V IPC与Linux有差异
在Linux上可以使用以下系统调用:
  msgget() - 创建/获取消息队列
  msgsnd() - 发送消息
  msgrcv() - 接收消息
  msgctl() - 控制消息队列

工作流程:
1. Server创建消息队列(key=75)，等待类型为1的请求
2. Client发送类型为1的消息(包含自身PID)
3. Server接收后显示'serving for client'和PID
4. Server发送应答消息(类型=client PID，内容=server PID)
5. Client接收后显示'receive reply from'和server PID
```

## 五、编程中遇到的困难及解决方法（特别是Mac下IPC兼容性问题）

### 5.1 遇到的困难

1. **Mac与Linux的IPC差异**：
   - Mac不支持System V IPC（msgget、shmget等）或需要特殊配置
   - 共享内存需要使用POSIX共享内存（shm_open）
   - 头文件和函数签名略有不同

2. **共享内存权限问题**：
   - shm_open需要正确的权限设置
   - 需要使用shm_unlink删除共享内存对象

3. **同步机制**：
   - 共享内存需要额外同步
   - 实验中使用sleep()简化，实际应用中应使用信号量

### 5.2 解决方法

1. **使用POSIX IPC替代System V IPC**：
   - 共享内存：shm_open + mmap
   - 消息队列：提供演示说明，指出Linux与Mac的差异

2. **权限处理**：
   - 设置0666权限，确保读写权限
   - 程序结束时调用shm_unlink清理

3. **同步简化**：
   - 使用wait()和sleep()实现简单同步
   - 在实验报告中说明实际应用中应使用信号量

### 5.3 实习心得

通过本次实验，深入理解了三种主要IPC机制的工作原理。

管道是最简单的IPC方式，适合父子进程间的简单数据传输。消息队列提供了更灵活的通信方式，可以按类型接收消息。共享内存是最快的IPC方式，因为进程直接访问内存，但需要额外的同步机制。

Mac与Linux的IPC差异让我认识到可移植性的重要性。POSIX IPC比System V IPC更现代，也更易于跨平台使用。在实际开发中，需要根据目标平台选择合适的IPC机制。

实验中使用的简化同步方式（sleep和wait）在实际应用中是不够的。真正的同步应该使用信号量或互斥锁，以避免竞态条件。

建议在后续实验中可以在Linux系统上进行，这样可以完整实现消息队列功能，并学习使用信号量进行同步。
