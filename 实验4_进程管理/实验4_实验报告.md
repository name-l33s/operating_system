# 实验4 进程管理

## 一、课程设计题目

实验4 进程管理

## 二、程序功能及设计思路

### 2.1 程序功能

本实验实现了一个进程管理系统，包含以下功能：
1. 父进程创建管道，用于子进程间通信
2. 父进程创建两个子进程
3. 子进程1每隔1秒通过管道发送消息
4. 子进程2从管道读取并显示消息
5. 父进程捕获SIGINT信号（Ctrl+C），向子进程发送终止信号
6. 子进程收到信号后输出提示信息并退出
7. 父进程等待子进程结束后释放管道并退出

### 2.2 进程创建、管道通信、信号处理原理

**1. 进程创建（fork）**
- `fork()`创建一个与父进程几乎完全相同的子进程
- 父进程返回子进程PID，子进程返回0
- 父子进程有独立的地址空间，但共享文件描述符

**2. 管道通信（pipe）**
- 管道是一种半双工通信方式，数据只能单向流动
- 管道有两个文件描述符：一个用于读，一个用于写
- 数据写入管道后，按先进先出顺序读取

**3. 信号处理（signal/kill）**
- `signal()`注册信号处理函数
- `kill()`向指定进程发送信号
- 常见信号：SIGINT（中断）、SIGUSR1（用户自定义信号）

### 2.3 父子进程并发执行关系和同步机制

```
父进程
  ├─ 创建管道
  ├─ 注册SIGINT处理函数
  ├─ fork() → 子进程1
  │         └─ 每隔1秒向管道写消息
  ├─ fork() → 子进程2
  │         └─ 从管道读消息并显示
  └─ wait() 等待子进程结束
```

同步机制：
- 管道提供了自然的同步：读操作会阻塞直到有数据
- 信号用于进程间异步通知

## 三、数据结构及算法设计

### 3.1 pipe管道的数据流向设计

```
子进程1 → write → 管道缓冲区 → read → 子进程2
```

- 子进程1关闭管道读端，只写
- 子进程2关闭管道写端，只读
- 父进程关闭管道两端，不使用管道

### 3.2 fork父子进程的分支控制

```c
pid_t pid = fork();
if (pid < 0) {
    // 错误处理
} else if (pid == 0) {
    // 子进程代码
} else {
    // 父进程代码
}
```

### 3.3 signal信号捕捉与kill信号发送流程

1. **父进程**：
   - 注册`SIGINT`处理函数
   - 收到`SIGINT`后，向子进程发送`SIGUSR1`
   - 等待子进程结束后退出

2. **子进程1**：
   - 注册`SIGUSR1`处理函数
   - 收到信号后输出提示并退出

3. **子进程2**：
   - 注册`SIGUSR1`处理函数
   - 收到信号后输出提示并退出

## 五、编程中遇到的困难及解决方法

### 5.1 遇到的困难

1. **信号处理函数安全**：信号处理函数中只能使用异步信号安全的函数
2. **管道关闭时机**：需要正确关闭不需要的管道端，否则可能导致读操作不返回
3. **子进程PID管理**：需要保存子进程PID，以便发送信号
4. **Mac系统信号处理**：Mac系统的信号处理与Linux有细微差异

### 5.2 解决方法

1. 信号处理函数中只调用`printf()`和`exit()`，这两个函数在大多数系统上是安全的
2. 子进程立即关闭不需要的管道端，父进程在fork后也关闭两端
3. 使用全局变量保存子进程PID，在信号处理函数中访问
4. 使用标准C库函数和POSIX标准调用，确保在Mac上兼容

### 5.3 实习心得

通过本次实验，深入理解了进程创建、管道通信和信号处理的工作原理。

fork()函数非常有趣，一次调用返回两次，在父子进程中执行不同的代码分支。管道提供了一种简单有效的进程间通信方式，虽然半双工，但对于单向数据流动已经足够。

信号是一种异步通信机制，用于通知进程发生了某个事件。实验中使用SIGINT作为触发信号，SIGUSR1作为自定义终止信号，展示了信号的灵活性。

进程管理是操作系统的核心功能，本次实验让我对并发编程有了更直观的认识。多个进程同时运行，通过管道和信号协调工作，这正是现代操作系统的典型工作方式。

建议在后续实验中可以增加共享内存、消息队列等其他IPC机制的实现，进一步学习进程间通信的各种方式。
